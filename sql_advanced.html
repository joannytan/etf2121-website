3  Advanced SQL

In the previous chapter, you learned how to query a single table — filtering, sorting, grouping, and summarising data. That is a solid foundation. But in the real world, data is never conveniently packed into one table. A company’s database might have dozens of tables, each storing a different piece of the puzzle. Advanced SQL is how you put those pieces together.

In this chapter, we will cover:

  • JOINs — combining data from multiple tables
  • CASE WHEN — conditional logic inside a query
  • Subqueries — queries inside queries
  • String Functions — cleaning and transforming text
  • Date Functions — working with dates and times
  • UNION — stacking result sets on top of each other

All examples use the FreshMart database — a fictional Australian supermarket chain with customers, products, stores, and transaction records.

3.2 JOINs

A JOIN combines rows from two tables based on a related column. The Transactions table stores CustomerID and StoreID — numbers that point to records in the Customers and Stores tables. On its own, a table full of ID numbers is not very useful. JOINs let you pull in the actual names and details from the other tables.

The general syntax is:

SELECT t1.column1, t2.column2, ...
FROM table1 AS t1
JOIN table2 AS t2
  ON t1.shared_column = t2.shared_column;

The ON clause specifies the condition that links the two tables — usually a foreign key in one table matched to a primary key in another. You must qualify column names with the table alias (e.g., T.CustomerID instead of just CustomerID) when the same column name exists in multiple tables.

3.2.1 INNER JOIN

An INNER JOIN returns only the rows that have a matching value in both tables. If a row has no match in the other table, it is excluded entirely.

SELECT T.TransactionID,
       C.FirstName,
       C.LastName,
       S.StoreName,
       T.TransactionDate,
       T.TotalAmount
FROM Transactions AS T
INNER JOIN Customers AS C
  ON T.CustomerID = C.CustomerID
INNER JOIN Stores AS S
  ON T.StoreID = S.StoreID;

This query joins all three tables at once. The result replaces the ID numbers with actual names:

3.2.2 LEFT JOIN

A LEFT JOIN returns all rows from the left table, plus the matching rows from the right table. Where there is no match, the right table’s columns come back as NULL.

This is the most useful JOIN for analysts. Use it when you want to see everything from one table regardless of whether it has matching records in another — for example, to find customers who have never made a purchase.

Show all customers, along with any transactions they have made:

SELECT C.FirstName, C.LastName, T.TransactionID, T.TotalAmount
FROM Customers AS C
LEFT JOIN Transactions AS T
  ON C.CustomerID = T.CustomerID;

A common pattern is to filter on NULL after a LEFT JOIN to find records with no match at all:

-- Customers who have never made a transaction
SELECT C.FirstName, C.LastName
FROM Customers AS C
LEFT JOIN Transactions AS T
  ON C.CustomerID = T.CustomerID
WHERE T.TransactionID IS NULL;

This returns only Nina Patel. She is presumably still weighing up her options in the breakfast cereal aisle.

A RIGHT JOIN is the mirror image of LEFT JOIN: it returns all rows from the right table, plus matching rows from the left table. Unmatched left-table rows become NULL.

FROM Transactions AS T
RIGHT JOIN Stores AS S
  ON T.StoreID = S.StoreID;

TipLEFT JOIN vs RIGHT JOIN — A Practical Tip

Most analysts write everything as a LEFT JOIN and simply swap the table order when needed. A RIGHT JOIN of A → B returns the same result as a LEFT JOIN of B → A. Sticking to LEFT JOIN throughout makes queries easier to read.

-- These two queries return the same result:

-- RIGHT JOIN version
SELECT S.StoreName, T.TransactionID
FROM Transactions AS T
RIGHT JOIN Stores AS S
  ON T.StoreID = S.StoreID;

-- Equivalent LEFT JOIN version (just swap the tables)
SELECT S.StoreName, T.TransactionID
FROM Stores AS S
LEFT JOIN Transactions AS T
  ON S.StoreID = T.StoreID;

3.2.4 JOIN Summary

Type What it returns
INNER JOIN Only rows with a match in both tables
LEFT JOIN All rows from the left table; NULL for unmatched right rows
RIGHT JOIN All rows from the right table; NULL for unmatched left rows

3.3 CASE WHEN

CASE WHEN is SQL’s version of if-else logic. It lets you create new calculated columns based on conditions — classifying values, creating labels, flagging rows — all inside a single query without touching the raw data.

SELECT column1,
       CASE
           WHEN condition1 THEN result1
           WHEN condition2 THEN result2
           ELSE default_result
       END AS new_column_name
FROM table_name;

SQL checks each WHEN condition from top to bottom and returns the first one that is true. If none match, the ELSE value is used. If there is no ELSE and nothing matches, the result is NULL.

Classify customers into spend tiers based on their lifetime TotalSpend:

SELECT FirstName, LastName, TotalSpend,
       CASE
           WHEN TotalSpend >= 5000 THEN 'High Spender'
           WHEN TotalSpend >= 1000 THEN 'Mid Spender'
           ELSE 'Light Spender'
       END AS SpendTier
FROM Customers;
FirstName LastName TotalSpend SpendTier
Priya Sharma 8420.50 High Spender
Jake Nguyen 340.00 Light Spender
Bev O’Brien 3150.75 Mid Spender
Marcus Lee 12300.00 High Spender
Amara Diallo 1890.25 Mid Spender
Gary McSnarl 520.00 Light Spender
Nina Patel 0.00 Light Spender

Nina Patel is officially a Light Spender with zero dollars spent. Her classification is technically accurate. Gary McSnarl spent $520 before churning — apparently he found cheaper milk elsewhere and never looked back.

You can also combine MembershipTier and TotalSpend for a more nuanced classification:

SELECT FirstName, LastName, MembershipTier, TotalSpend,
       CASE
           WHEN MembershipTier = 'Gold'   AND TotalSpend >= 5000 THEN 'Loyal VIP'
           WHEN MembershipTier = 'Gold'   AND TotalSpend <  5000 THEN 'Gold Underperformer'
           WHEN MembershipTier = 'Silver'                        THEN 'Growing'
           WHEN MembershipTier = 'Bronze' AND Churned = 1        THEN 'Lost'
           ELSE 'New'
       END AS CustomerSegment
FROM Customers;

You can also use CASE WHEN inside aggregate functions. This pattern is used constantly for pivot-style summaries:

-- Count churned vs active customers in a single query
SELECT
    SUM(CASE WHEN Churned = 1 THEN 1 ELSE 0 END) AS ChurnedCount,
    SUM(CASE WHEN Churned = 0 THEN 1 ELSE 0 END) AS ActiveCount
FROM Customers;

3.4 Subqueries

A subquery is a query nested inside another query. The inner query runs first, and its result is handed to the outer query to use. This lets you answer two-step questions: “first calculate this, then use that result to filter or compute something else.”

Subqueries appear in three places:

  • Inside WHERE — to filter using a calculated value
  • Inside FROM — to create a temporary derived table
  • Inside SELECT — to add a per-row calculated column

3.4.1 Subqueries in WHERE

Find customers whose lifetime TotalSpend is above the average:

SELECT FirstName, LastName, TotalSpend
FROM Customers
WHERE TotalSpend > (SELECT AVG(TotalSpend) FROM Customers);

The inner query runs first and returns the average spend (approximately $3,803). The outer query then uses that value to filter rows.

FirstName LastName TotalSpend
Priya Sharma 8420.50
Marcus Lee 12300.00

Marcus Lee and Priya Sharma are the only two customers above average — which is less surprising when you notice that Marcus spent $210 and $190 in a single month. He is clearly shopping for a village.

Find transactions where the basket total exceeds the average transaction amount:

SELECT TransactionID, TotalAmount
FROM Transactions
WHERE TotalAmount > (SELECT AVG(TotalAmount) FROM Transactions);

3.4.2 Subqueries in FROM

A subquery in the FROM clause creates a temporary derived table that the outer query treats like a real table. You must give it an alias.

Calculate the average transaction amount per state by first joining transactions to stores, then grouping in the outer query:

SELECT State, AVG(TotalAmount) AS AvgTransactionAmount
FROM (
    SELECT T.TotalAmount, S.State
    FROM Transactions AS T
    INNER JOIN Stores AS S
      ON T.StoreID = S.StoreID
) AS TransactionsByState
GROUP BY State;

The inner query builds a table that links each transaction to the store’s state. The outer query then groups and aggregates it. In SQL Server, you cannot reference a column alias created in SELECT directly inside GROUP BY — wrapping it in a subquery solves this cleanly.

3.4.3 Subqueries in SELECT

A subquery in the SELECT clause adds a calculated value to every row. It must return exactly one value (a scalar).

Show each transaction alongside the overall average transaction amount and the difference:

SELECT TransactionID, TotalAmount,
       (SELECT AVG(TotalAmount) FROM Transactions) AS AvgAmount,
       TotalAmount - (SELECT AVG(TotalAmount) FROM Transactions) AS DifferenceFromAvg
FROM Transactions;

3.5 String Functions

Real data is messy. Names have inconsistent capitalisation. Text fields have trailing spaces. Values need reformatting before they are useful. String functions let you clean and reshape text directly inside SQL.

All examples use SQL Server syntax.

3.5.1 UPPER() and LOWER()

Convert text to all uppercase or all lowercase. This is important for consistent comparisons — 'Victoria' and 'VICTORIA' are treated as different values by SQL unless you normalise first.

SELECT FirstName, LastName,
       UPPER(LastName)  AS LastNameUpper,
       LOWER(FirstName) AS FirstNameLower
FROM Customers;

A common use: case-insensitive search by converting both the column and the search term to the same case.

SELECT ProductName
FROM Products
WHERE LOWER(Category) LIKE '%dairy%';

3.5.2 LEN()

Returns the number of characters in a string. SQL Server uses LEN() — note this is different from LENGTH() used in MySQL and SQLite.

SELECT ProductName, LEN(ProductName) AS NameLength
FROM Products;

Find products with unusually long names:

SELECT ProductName, LEN(ProductName) AS NameLength
FROM Products
WHERE LEN(ProductName) > 15;

3.5.3 CONCAT()

Joins two or more strings together into one.

SELECT CONCAT(FirstName, ' ', LastName) AS FullName,
       Suburb,
       State
FROM Customers;
FullName Suburb State
Priya Sharma Fitzroy VIC
Jake Nguyen Parramatta NSW
Bev O’Brien Toowoomba QLD

In SQL Server you can also use the + operator to concatenate. However, you cannot use + to join a number directly to a string — you must convert the number to text first using CAST():

SELECT FirstName + ' ' + LastName + ' has spent $' + CAST(TotalSpend AS VARCHAR) AS SpendSummary
FROM Customers;

3.5.4 SUBSTRING(), LEFT(), RIGHT()

SUBSTRING(string, start, length) extracts part of a string. Positions are 1-indexed — the first character is at position 1.

SELECT ProductName, SUBSTRING(ProductName, 1, 10) AS FirstTenChars
FROM Products;

LEFT(string, n) and RIGHT(string, n) are convenient shortcuts for extracting from the beginning or end:

SELECT FirstName,
       LEFT(FirstName, 3)  AS FirstThree,
       RIGHT(LastName, 3)  AS LastThree
FROM Customers;

3.5.5 TRIM(), LTRIM(), RTRIM()

Removes leading and/or trailing spaces. Invisible extra spaces are a common source of silent bugs — a value that looks like 'Fitzroy' in a report might actually be ' Fitzroy' in the database, causing WHERE Suburb = 'Fitzroy' to silently return no rows.

SELECT TRIM(StoreName)   AS BothSidesTrimmed   -- removes spaces on both sides
FROM Stores;

SELECT LTRIM(StoreName)  AS LeftTrimmed         -- removes leading spaces only
FROM Stores;

SELECT RTRIM(StoreName)  AS RightTrimmed        -- removes trailing spaces only
FROM Stores;

3.5.6 REPLACE()

Replaces every occurrence of a substring with another string. Useful for fixing consistent data entry errors or abbreviating labels.

-- Abbreviate store names in a summary report
SELECT StoreName,
       REPLACE(StoreName, 'FreshMart', 'FM') AS ShortName
FROM Stores;
StoreName ShortName
FreshMart Fitzroy FM Fitzroy
FreshMart Parramatta FM Parramatta
FreshMart Toowoomba FM Toowoomba
FreshMart Carlton FM Carlton
FreshMart Adelaide CBD FM Adelaide CBD
FreshMart Darwin FM Darwin

The rebranding team is going to love this.

3.6 Date Functions

Almost every real-world dataset has dates — join dates, transaction records, store opening dates. Being able to extract parts of dates, calculate durations, and filter by time period is an essential analyst skill. All examples use SQL Server syntax.

3.6.1 GETDATE()

Returns the current date and time from the database server.

SELECT GETDATE() AS CurrentDateTime;

Useful as a reference point for “how long ago” calculations.

3.6.2 YEAR(), MONTH(), DAY()

Extract individual components from a date column.

SELECT TransactionID,
       YEAR(TransactionDate)  AS TxnYear,
       MONTH(TransactionDate) AS TxnMonth,
       DAY(TransactionDate)   AS TxnDay
FROM Transactions;

Filter to transactions from February 2024 only:

SELECT TransactionID, TotalAmount
FROM Transactions
WHERE YEAR(TransactionDate) = 2024
  AND MONTH(TransactionDate) = 2;

Calculate approximate years since each customer joined:

SELECT FirstName, LastName, YearJoined,
       YEAR(GETDATE()) - YearJoined AS YearsAsMember
FROM Customers;
FirstName LastName YearJoined YearsAsMember
Priya Sharma 2018 8
Jake Nguyen 2023 3
Bev O’Brien 2020 6
Marcus Lee 2017 9
Amara Diallo 2021 5
Gary McSnarl 2022 4
Nina Patel 2024 2

(Approximate values based on 2026.)

3.6.3 DATEDIFF()

Calculates the difference between two dates. Syntax: DATEDIFF(unit, start_date, end_date).

Common units: DAY, MONTH, YEAR.

How many days ago did each transaction take place?

SELECT TransactionID, TransactionDate,
       DATEDIFF(DAY, TransactionDate, GETDATE()) AS DaysSinceTransaction
FROM Transactions;

How many months have passed since FreshMart Darwin opened?

SELECT DATEDIFF(MONTH, '2024-01-01', GETDATE()) AS MonthsSinceOpening;
TipStoring Years as Integers vs Dates

In the Customers table, YearJoined is stored as an INT (e.g., 2018) rather than a full DATE. This works fine for year-level calculations using YEAR(GETDATE()) - YearJoined, but it does not work with DATEDIFF() directly. In a production database, you would store join dates as a proper DATE column for full date arithmetic.

3.6.4 DATEADD()

Adds (or subtracts) a time interval from a date. Syntax: DATEADD(unit, number, date). Use a negative number to subtract.

What would each transaction date be if a promotional reminder was sent 14 days later?

SELECT TransactionID, TransactionDate,
       DATEADD(DAY, 14, TransactionDate) AS ReminderDate
FROM Transactions;

What was the date 3 months before each transaction (to identify pre-transaction browsing windows)?

SELECT TransactionID, TransactionDate,
       DATEADD(MONTH, -3, TransactionDate) AS ThreeMonthsBefore
FROM Transactions;

3.6.5 FORMAT()

Formats a date as a human-readable string. Useful for reports and dashboards where raw date formats look unfriendly.

3.7 UNION

UNION stacks the results of two separate SELECT queries into one combined result set by appending the rows of the second query below the rows of the first.

Three rules must be followed:

  1. Both queries must return the same number of columns.
  2. The columns must have compatible data types, in the same order.
  3. The column names in the final output come from the first query.

3.7.1 UNION vs UNION ALL

UNION UNION ALL
Duplicates Removed Kept
Performance Slower (must scan for duplicates) Faster
Use when You need a deduplicated list You want all rows including repeats

Default to UNION ALL unless you specifically need duplicate removal.

3.7.2 Example: Active and At-Risk Customers

The marketing team wants a single report showing both active and churned customers, labelled clearly. Two separate pools of data, same structure — a perfect job for UNION ALL.

SELECT FirstName, LastName, MembershipTier, TotalSpend, 'Active' AS Status
FROM Customers
WHERE Churned = 0

UNION ALL

SELECT FirstName, LastName, MembershipTier, TotalSpend, 'Churned' AS Status
FROM Customers
WHERE Churned = 1;

Both queries return five columns with compatible types. You can add ORDER BY at the very end to sort the full combined result:

SELECT FirstName, LastName, MembershipTier, TotalSpend, 'Active' AS Status
FROM Customers
WHERE Churned = 0

UNION ALL

SELECT FirstName, LastName, MembershipTier, TotalSpend, 'Churned' AS Status
FROM Customers
WHERE Churned = 1

ORDER BY TotalSpend DESC;

3.7.3 Example: Deduplication with UNION

UNION (without ALL) removes duplicates across both result sets. For example, to get a deduplicated list of all states that have either a FreshMart store or a VIC/NSW customer:

SELECT State FROM Stores
UNION
SELECT State FROM Customers WHERE State IN ('VIC', 'NSW');

Each state appears exactly once, even if it appears in both tables. In practice, SELECT DISTINCT is more natural for deduplication within a single table, but UNION is essential when deduplicating across separate queries or separate tables.

TipWhen to Use UNION

Use UNION when you have two separate pools of data with the same structure that you want to analyse together:

  • Combining transaction data from two financial years stored in separate tables
  • Merging customer records from an old system and a new system during a migration
  • Building a report that shows “top 3 biggest spenders” and “bottom 3 spenders” in one list

3.8 Putting It All Together

Real-world queries rarely use just one technique. Here is a query that combines several tools from this chapter to answer a practical analytical question.

Question: For each customer who has made at least one transaction, show their full name, state, total amount spent across all transactions, number of transactions, spend tier, and whether their total transaction spend is above or below the average.

SELECT
    CONCAT(C.FirstName, ' ', C.LastName) AS CustomerName,
    C.State,
    SUM(T.TotalAmount)                   AS TotalTransactionSpend,
    COUNT(T.TransactionID)               AS NumberOfTransactions,
    CASE
        WHEN SUM(T.TotalAmount) >= 200 THEN 'High Spender'
        WHEN SUM(T.TotalAmount) >= 80  THEN 'Mid Spender'
        ELSE 'Light Spender'
    END AS SpendTier,
    CASE
        WHEN SUM(T.TotalAmount) > (SELECT AVG(sub.CustomerTotal)
                                   FROM (SELECT CustomerID, SUM(TotalAmount) AS CustomerTotal
                                         FROM Transactions
                                         GROUP BY CustomerID) AS sub)
        THEN 'Above Average'
        ELSE 'Below Average'
    END AS SpendVsAverage
FROM Customers AS C
INNER JOIN Transactions AS T
  ON C.CustomerID = T.CustomerID
GROUP BY C.CustomerID, C.FirstName, C.LastName, C.State
ORDER BY TotalTransactionSpend DESC;

This single query uses INNER JOIN, CASE WHEN, a subquery, aggregate functions, GROUP BY, and ORDER BY. If you can read and write a query like this, you are thinking like a data analyst.

You have now covered the full essential SQL toolkit for business data analysis. JOINs connect your tables. CASE WHEN classifies and labels your data. Subqueries let you compute in layers. String and date functions clean and enrich what you retrieve. UNION combines result sets from multiple directions.
