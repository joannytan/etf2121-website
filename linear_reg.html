9  Week 7: Linear Regression

Part 3 · Week 7

Linear Regression

The Head of Marketing has a question: "Can we predict how much a new customer will spend over their lifetime — based on how they behave in their first weeks?" If we can, we can prioritise which new sign-ups deserve a special welcome offer. This chapter builds your first predictive model.

📦 tidyverse 📦 tidymodels

9.1 7.1 From Description to Prediction

In Chapter 6 we described the FreshMart customers: we found that customers who visit more often tend to spend more. But description only tells us about the past. Prediction asks: given what we know about a new customer today, what will they likely do in the future?

A regression model is a mathematical equation that uses one or more input variables (called predictors or features) to produce a predicted numeric output (called the outcome or target).

For FreshMart, we want to predict:

\[\text{TotalSpend} = f(\text{TotalTransactions}, \text{DaysSinceFirstPurchase}, \text{City}, \ldots)\]

The simplest version of this is linear regression — which assumes the relationship between predictors and outcome is a straight line.

9.2 7.2 Simple Linear Regression

9.2.1 The model equation

Simple linear regression uses one predictor to predict the outcome. The model is:

\[\hat{y} = \beta_0 + \beta_1 x\]

Where:

  • \(\hat{y}\) is the predicted value of the outcome (TotalSpend)
  • \(x\) is the predictor (e.g., TotalTransactions)
  • \(\beta_0\) is the intercept — the predicted \(\hat{y}\) when \(x = 0\)
  • \(\beta_1\) is the slope — how much \(\hat{y}\) changes for each 1-unit increase in \(x\)

The model finds the values of \(\beta_0\) and \(\beta_1\) that make the line fit the data as closely as possible. “As closely as possible” means minimising the sum of squared residuals — the squared differences between actual and predicted values.

Note📚 Recall: The Line of Best Fit

In first-year statistics you encountered lines of best fit in the context of scatter plots. Linear regression is exactly that — but now we use it to make numerical predictions and we measure how good the line is formally.

The residual for observation \(i\) is: \(e_i = y_i - \hat{y}_i\) (actual minus predicted). A good model has small residuals.

The animation below sweeps through many candidate lines. Each orange segment is a residual. Watch how the SSE (sum of squared residuals) in the title changes — the green line at the minimum SSE is exactly what linear regression finds automatically.

9.2.2 Setting up — load packages and data

Run this block first. It loads tidymodels and the FreshMart dataset, and creates the training/test split we’ll need throughout the chapter.

NoteWhy split the data?

We split the dataset into a training set and a test set. The model is fitted on the training set only. The test set is held out until the very end — it represents “new customers the model has never seen” and gives us an honest estimate of how well the model will perform in the real world.

set.seed(42) makes the random split reproducible — you’ll get the same split every time you run this code.

9.2.3 The tidymodels workflow

tidymodels uses a consistent three-step recipe for building any model:

1. Specify   →  What type of model? What engine?
2. Fit       →  Train the model on the training data
3. Evaluate  →  Check performance on the test data

This might seem like extra work compared to base R’s lm(), but the big advantage is that every model in this course — logistic regression, trees, KNN — follows exactly the same three steps. Change one line and you change the model type.

9.2.4 Fitting a simple linear regression

The tidy() function from the broom package (part of tidymodels) returns the model results as a clean tibble. Let’s read the output:

Column Meaning
term Which coefficient (intercept, or the predictor name)
estimate The estimated value of \(\beta_0\) or \(\beta_1\)
std.error How uncertain we are about that estimate
statistic The t-statistic for the hypothesis test
p.value Is this coefficient statistically different from zero?

Reading the output:

  • (Intercept) ≈ some value — this is the predicted spend for a customer with zero transactions
  • TotalTransactions coefficient ≈ some positive value — for each additional transaction, this is how much more we predict the customer will spend
Note📚 Recall: p-values

From first-year statistics: a p-value measures the probability of seeing a result as extreme as we observed, if there were truly no relationship. Convention: if p < 0.05, we say the result is “statistically significant” — i.e., the predictor appears to genuinely influence the outcome (rather than just by random chance).

9.2.5 The model equation and predictions

Important✏️ Concept Check — Section 7.2

Q1. In the fitted equation TotalSpend = 25.60 + 18.40 × TotalTransactions, what does the slope of 18.40 mean in plain English?

Q2. A customer has made 10 transactions. Using the equation above, what is their predicted total spend?

Q3. The p-value for TotalTransactions is 0.000 (effectively zero). What does this tell you?

Tip✅ Answers

A1. For every additional transaction a customer makes, their predicted total lifetime spend increases by $18.40, on average — holding everything else constant.

A2. $25.60 + 18.40 × 10 = $25.60 + $184.00 = $209.60

A3. A p-value of essentially zero means the probability of seeing a slope this large just by chance (if there were no real relationship) is effectively zero. We have very strong statistical evidence that TotalTransactions genuinely predicts TotalSpend.

9.3 7.3 Model Fit: How Good Is Our Line?

9.3.1 R² — the proportion of variance explained

glance() gives overall model statistics:

The key statistic is R² (r-squared): the proportion of variation in TotalSpend that is explained by the model. R² ranges from 0 to 1:

  • R² = 0: the model explains nothing (no better than guessing the mean every time)
  • R² = 1: the model explains everything perfectly (every prediction is exact)
  • R² = 0.50: the model explains half the variation in TotalSpend

9.3.2 RMSE — how big are our errors?

R² tells us about relative fit. The Root Mean Squared Error (RMSE) tells us the average size of our errors in the same units as the outcome ($AUD):

\[\text{RMSE} = \sqrt{\frac{1}{n}\sum_{i=1}^{n}(y_i - \hat{y}_i)^2}\]

An RMSE of, say, $150 means our predictions are off by about $150 on average. Whether that’s acceptable depends on the business context — good enough to prioritise customer outreach, perhaps not good enough to set individual credit limits.

9.3.3 Residual plot

A residual plot checks whether the errors are random (good) or have a pattern (bad). We want residuals scattered randomly around zero with no trend.

Note

What to look for: If the residuals fan out as predictions increase (a “funnel” shape), this suggests the model is less accurate for high-spending customers. This is common with right-skewed outcome variables like TotalSpend.

Important✏️ Concept Check — Section 7.3

Q1. R² = 0.52 on the test set. Describe what this means in plain English.

Q2. RMSE = $210 and the average TotalSpend is $414. Is this a good result? How would you decide?

Q3. You look at the residual plot and see that all the large positive residuals (model under-predicted) are from Melbourne customers. What does this suggest?

Tip✅ Answers

A1. The model explains 52% of the variation in customer total spend. The remaining 48% is driven by things our single predictor doesn’t capture — city, average transaction value, tenure, or purely random differences between customers.

A2. This is contextual. RMSE of $210 on an average spend of $414 means a typical error of about 50% of the average value — quite large. However, for a model with just one predictor, it might be a reasonable starting point. We’d compare against a baseline (e.g., “what if we just always predicted the mean spend?”) and against a model with more predictors.

A3. It suggests our model is systematically under-predicting spend for Melbourne customers — there’s something about Melbourne that our predictor doesn’t capture. Adding City as a predictor (which we’ll do next) should fix this.

9.4 7.4 Multiple Linear Regression

A model with just one predictor often misses important information. Multiple regression uses several predictors simultaneously. The equation becomes:

\[\hat{y} = \beta_0 + \beta_1 x_1 + \beta_2 x_2 + \beta_3 x_3 + \ldots\]

Each coefficient \(\beta_j\) shows the effect of predictor \(x_j\) on the outcome while holding all other predictors constant.

9.4.1 The recipe approach

When we have multiple predictors — especially categorical ones like City — we use a recipe to preprocess the data before fitting. A recipe describes transformations to apply to the predictors.

City is a categorical variable (text). Most statistical models need numeric inputs, so we convert it to dummy variables: one binary column per city (1 = that city, 0 = not).

Notice that City has been replaced by several binary columns: City_Darwin, City_Melbourne, etc. (one city is omitted as the “baseline” to avoid redundancy — this is called the reference category).

9.4.2 Building the workflow

Interpreting the coefficients:

  • TotalTransactions: for each additional visit, predicted spend increases by $X — holding all other predictors constant
  • DaysSinceFirstPurchase: for each additional day of tenure, spend increases by $Y
  • City_Melbourne: customers in Melbourne spend $Z more (or less if negative) than the reference city — on average, given the same transactions and tenure
Note📦 R Practice — Exercise 7.4.1

The fm_recipe uses TotalTransactions, AvgTransactionValue, DaysSinceFirstPurchase, and City as predictors. Try removing AvgTransactionValue from the recipe. Does the model’s R² increase or decrease? What does this tell you?

Tip✅ Solution 
fm_recipe_v2 <- recipe(
  TotalSpend ~ TotalTransactions + DaysSinceFirstPurchase + City,
  data = fm_train
) |>
  step_dummy(City)

lm_v2 <- workflow() |>
  add_recipe(fm_recipe_v2) |>
  add_model(lm_spec) |>
  fit(data = fm_train)

glance(lm_v2)

If R² decreases, it means AvgTransactionValue was contributing useful predictive information. If it barely changes, the variable wasn’t adding much beyond what TotalTransactions already captures.

Important✏️ Concept Check — Section 7.4

Q1. In multiple regression, the coefficient for TotalTransactions is interpreted “holding all other predictors constant.” Why is this phrase important?

Q2. After adding City to the model, R² increases from 0.52 to 0.58. Is this extra complexity worth it?

Q3. Why do we use dummy variables for City rather than coding it as 1=Sydney, 2=Melbourne, 3=Brisbane, etc.?

Tip✅ Answers

A1. Without this phrase, we might confuse the individual effect of transactions with the combined effect of transactions and other variables. In multiple regression, each coefficient isolates the effect of one predictor, assuming the others don’t change. This is what makes the coefficients interpretable.

A2. It depends. An increase of 0.06 in R² is modest — the model explains 6% more variation. Whether this is worth the added complexity (and harder-to-explain coefficients) is a judgement call. In practice, we’d also check whether the improvement holds on the test set, not just training.

A3. If we coded Sydney = 1, Melbourne = 2, Brisbane = 3, the model would treat this as a numeric variable — implying that Melbourne is “twice Sydney” and Brisbane is “three times Sydney.” That makes no mathematical sense. Dummy variables avoid this by treating each city as a separate binary yes/no.

9.5 7.5 Making Predictions

Once a model is fitted, we can use it to predict outcomes for new, unseen customers.

9.5.1 Evaluating on the test set

This is the most important evaluation — the test set simulates real “new customers” the model has never seen.

Points close to the diagonal line are accurate predictions. Points far from the line are cases where our model struggles — notice the high-spending customers (top-right area) where the model tends to under-predict.

Important✏️ Concept Check — Section 7.5

Q1. Why do we evaluate model performance on the test set rather than the training set?

Q2. The actual-vs-predicted plot shows the model consistently under-predicts for customers with spend above $2,000. What could explain this?

Q3. A colleague suggests: “Since the model was trained on all our existing customers, why not just retrain it every week with the latest data?” What is the benefit of this approach?

Tip✅ Answers

A1. Because the model was fitted on the training data — it has “seen” those rows. Evaluating on training data would be like giving a student the exam before the test and then measuring how well they do. The test set is genuinely unseen data, giving us an honest estimate of real-world performance.

A2. Several possibilities: (1) High spenders may have predictors (like very high transaction frequency or very high avg value) that are outside the range of the training data — the model is “extrapolating.” (2) The linear assumption may break down at high spend levels — the relationship may be non-linear. (3) There may be predictor variables we haven’t included (like product category preferences) that differentiate high spenders.

A3. The benefit is that the model stays up to date with changing customer behaviour. If customer patterns shift (e.g., post-COVID shopping habits change), a model trained on old data will drift — regular retraining keeps it current.

9.6 7.6 A Word on Model Assumptions

Linear regression makes several assumptions. Violating them doesn’t always break the model, but it can make the predictions unreliable or the p-values misleading. Here are the key ones — with FreshMart context:

1. Linearity. The relationship between each predictor and the outcome should be (approximately) linear. Our residual plot can reveal non-linearity (a curved pattern in the residuals).

2. Independence. Observations should be independent of each other. For FreshMart, this is reasonable — one customer’s spend doesn’t directly affect another’s.

3. Constant variance (homoscedasticity). The spread of residuals should be similar across all predicted values. If residuals fan out as predicted values increase (a funnel shape), this is a problem.

4. Normality of residuals. The residuals should be approximately normally distributed. This matters most for small samples and for the reliability of p-values.

NoteExercise 7.5 — Overfitting Check

You’ve been asked to compare a “simple” model (TotalTransactions only) vs a “complex” model (all predictors). Calculate both models’ R² on the training set and the test set. Present the results in a small table. Which model overfits more, and how can you tell?

Tip✅ Solution 
# Simple model
s_train <- augment(lm_simple, new_data = fm_train) |>
  metrics(truth = TotalSpend, estimate = .pred) |>
  filter(.metric == "rsq") |>
  mutate(model = "Simple", set = "Train")

s_test <- augment(lm_simple, new_data = fm_test) |>
  metrics(truth = TotalSpend, estimate = .pred) |>
  filter(.metric == "rsq") |>
  mutate(model = "Simple", set = "Test")

# Multiple model
m_train <- augment(lm_multi, new_data = fm_train) |>
  metrics(truth = TotalSpend, estimate = .pred) |>
  filter(.metric == "rsq") |>
  mutate(model = "Multiple", set = "Train")

m_test <- augment(lm_multi, new_data = fm_test) |>
  metrics(truth = TotalSpend, estimate = .pred) |>
  filter(.metric == "rsq") |>
  mutate(model = "Multiple", set = "Test")

bind_rows(s_train, s_test, m_train, m_test) |>
  select(model, set, .estimate) |>
  pivot_wider(names_from = set, values_from = .estimate)

A model that overfits shows a large gap between training and test R². If both gaps are small, the models are generalising well.
