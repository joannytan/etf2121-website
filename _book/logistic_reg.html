10  Logistic Regression

Part 3 · Week 8

Logistic Regression

Last week we predicted how much a customer would spend — a number. This week the marketing team needs a yes/no answer: will this customer become a high-value customer? That shift from predicting a number to predicting a category is what logistic regression is built for.

📦 tidyverse 📦 tidymodels
🔁 Recap — What we built last week

In Chapter 7 we used linear regression to predict TotalSpend — a continuous dollar amount. We learned the tidymodels workflow (specify → recipe → workflow → fit → evaluate), and assessed model quality using R² and RMSE on a held-out test set.

This week: same tidymodels structure, different type of outcome — a category instead of a number.

10.1 8.1 When the Outcome Is a Category

The Head of FreshMart’s loyalty team has a new question: “Can we predict whether a new customer will become high-value, based on their early behaviour?”

If we can flag likely high-value customers early, we can send them a targeted welcome offer before they lose interest. The answer isn’t a dollar amount — it’s one of two options: Yes (High Value) or No. This is a classification problem.

Our target variable is IsHighValue — TRUE if lifetime TotalSpend > $500, FALSE otherwise.

10.1.1 Why not use linear regression for this?

You might think: fit linear regression where the outcome is 1 (TRUE) or 0 (FALSE). The problem is that a straight line can produce predictions outside the 0–1 range — such as a “probability” of −0.3 or 1.7 — which is mathematically meaningless.

Logistic regression solves this by using a special S-shaped curve called the sigmoid function that squeezes all outputs into the range 0–1.

The model outputs a probability for each customer (e.g., 78% likely to be high value). We then apply a decision threshold — by default 0.5 — to convert that probability into a class label: ≥ 0.5 → predict TRUE, < 0.5 → predict FALSE.

10.1.2 The model equation and log-odds

The sigmoid is not an arbitrary choice — it emerges from the model structure. Logistic regression assumes the log-odds of the outcome is a linear function of the predictors:

\[\log\!\left(\frac{p}{1-p}\right) = \beta_0 + \beta_1 x_1 + \beta_2 x_2 + \cdots\]

The left side is the logit (log of the odds ratio). Solving for \(p\) gives the sigmoid directly:

\[p = P(\text{High Value} = 1 \mid \mathbf{x}) = \frac{1}{1 + e^{-(\beta_0 + \beta_1 x_1 + \beta_2 x_2 + \cdots)}}\]

So the same linear combination as linear regression is happening inside the exponent — logistic regression is a linear model in disguise, transformed to keep outputs between 0 and 1.

Reading the coefficients: Each \(\beta_j\) is on the log-odds scale. The more practical interpretation uses the odds ratio \(e^{\beta_j}\):

\(\beta_j\) Odds ratio \(e^{\beta_j}\) Meaning
Positive > 1 Higher \(x_j\) increases the probability of TRUE
Negative < 1 Higher \(x_j\) decreases the probability of TRUE
≈ 0 ≈ 1 \(x_j\) has negligible effect

For example, if the coefficient for TotalTransactions is 0.40, each additional transaction multiplies the odds of being High Value by \(e^{0.40} \approx 1.49\) — a 49% increase in odds, holding all other predictors constant.

How are coefficients estimated? Logistic regression uses maximum likelihood estimation (MLE), not OLS. The algorithm finds the \(\beta\) values that make the observed outcomes most probable:

\[\hat{\boldsymbol{\beta}} = \arg\max_{\boldsymbol{\beta}} \prod_{i:\,y_i=1} p_i \;\times\; \prod_{i:\,y_i=0}(1 - p_i)\]

In plain English: choose coefficients so the model assigns high probability to customers who actually were High Value, and low probability to those who weren’t. There is no closed-form solution — R’s glm() solves this iteratively via Newton-Raphson.

✏️ Concept Check — 8.1

Q1. A logistic model outputs 0.72 for a customer. Using the default threshold, what class do we predict?

Q2. Why can’t linear regression be used directly when the outcome is TRUE/FALSE?

Q3. Looking at the sigmoid curve: as the linear combination gets very large (far right), what does the predicted probability approach?

✅ Answers

A1. 0.72 > 0.5, so we predict TRUE (High Value).

A2. A straight line can produce predictions outside 0–1 (e.g., −0.3 or 1.4). These cannot be interpreted as probabilities. The logistic function guarantees valid probability outputs.

A3. It approaches 1.0 (100%) — the model becomes increasingly confident the customer is high-value as the predictors indicate stronger purchasing behaviour.

10.2 8.2 Fitting Logistic Regression in R

The tidymodels workflow is nearly identical to Chapter 7 — we swap linear_reg() for logistic_reg() and add set_mode("classification").

10.2.1 Setup

strata = IsHighValue

Stratified splitting ensures the training and test sets have roughly the same proportion of high-value customers. Without it, random chance might leave almost all high-value customers in one set.

10.2.2 Specify → Recipe → Workflow → Fit

Reading the output: The estimate column shows log-odds coefficients. You don’t need to work with log-odds directly — what matters is:

  • Positive coefficient → higher values of that predictor increase the probability of TRUE (High Value)
  • Negative coefficient → higher values decrease it
  • Small p-value (< 0.05) → that predictor is statistically significant
📦 R Practice — Exercise 8.2.1

Look at the tidy() output. Which predictor has the largest positive coefficient? What does this tell you about which variable most strongly predicts whether a customer becomes high-value?

✅ Solution

Find the row with the largest positive estimate. Because we used step_normalize(), the coefficients are on a comparable scale — the largest coefficient corresponds to the most influential predictor. Typically TotalTransactions or AvgTransactionValue will rank highest, meaning these best distinguish high-value from regular customers.

✏️ Concept Check — 8.2

Q1. What is the only line that changed between the Chapter 7 linear regression workflow and this logistic regression workflow?

Q2. We used step_normalize() in the recipe. What does this do, and why does it matter here?

Q3. The coefficient for DaysSinceFirstPurchase is positive. What does this mean in plain English?

✅ Answers

A1. The model specification changed from linear_reg() to logistic_reg() |> set_mode("classification"). The recipe, workflow, and fit() are structurally identical. This is the key benefit of tidymodels — swap one line to change the model type.

A2. step_normalize() rescales each numeric variable to have mean = 0 and SD = 1. This puts predictors measured in different units (e.g., days vs. dollars) on a comparable scale, so their coefficients can be meaningfully compared.

A3. Customers who have been with FreshMart longer are more likely to be classified as High Value — holding other predictors constant. This makes intuitive sense: longer tenure gives more time to accumulate spend.

10.3 8.3 Evaluating a Classifier: The Confusion Matrix

A single accuracy number can be deceiving. If 75% of customers are not high-value, a model that always predicts FALSE achieves 75% accuracy — without learning anything. We need to look inside the errors.

The four cells:

Predicted: Not High Value Predicted: High Value
Actual: Not High Value ✅ True Negative ❌ False Positive
Actual: High Value ❌ False Negative ✅ True Positive

For FreshMart’s goal (nurture high-value customers early), False Negatives are the costlier error — we miss a valuable customer. We’d prefer to occasionally send an offer to a regular customer (False Positive) than miss a future loyal one.

10.3.1 Accuracy, sensitivity, and specificity

Metric Meaning
Accuracy Overall fraction of correct predictions
Sensitivity Of all actual High-Value customers, what fraction did we catch?
Specificity Of all actual regular customers, what fraction did we correctly identify?

10.3.2 Adjusting the threshold and the ROC curve

By lowering the threshold below 0.5 we catch more high-value customers (higher sensitivity), at the cost of more false positives (lower specificity). The ROC curve shows this trade-off across every possible threshold at once.

AUC of 1.0 is perfect; 0.5 is random. Values above 0.80 are generally considered good.

✏️ Concept Check — 8.3

Q1. Our model has accuracy = 0.84 and the test set is 75% negative. What is the “naive baseline” accuracy, and how much better is our model?

Q2. We lower the threshold from 0.5 to 0.3 to catch more high-value customers. What happens to the number of False Positives?

Q3. What does AUC = 0.50 tell you about a model?

✅ Answers

A1. The naive baseline (always predict FALSE) achieves 75% accuracy. Our model at 84% beats it by 9 percentage points. Context makes a difference — 84% isn’t as impressive if we forget what the baseline is.

A2. False Positives increase. Lowering the threshold means more customers are flagged as High Value — including some regular customers who are misclassified. This is the sensitivity-specificity trade-off in action.

A3. A model with AUC = 0.50 is no better than random guessing — it has learned nothing useful from the predictors.

Exercise 8.1 — Predict for FreshMart Characters

Use the fitted model to predict IsHighValue for Marcus Lee, Gary McSnarl, Jake Nguyen, and Amara Diallo. Compare the model’s prediction to their actual IsHighValue status (from the dataset).

characters <- tibble(
  TotalTransactions      = c(48,    1,     7,     14),
  AvgTransactionValue    = c(256.25, 34.90, 48.57, 62.86),
  DaysSinceFirstPurchase = c(1868,  1026,  795,   146),
  City                   = c("Sydney", "Melbourne", "Melbourne", "Perth")
)
✅ Solution 
augment(lr_fit, new_data = characters) |>
  bind_cols(tibble(Name = c("Marcus Lee", "Gary McSnarl", "Jake Nguyen", "Amara Diallo"),
                  Actual = c("TRUE", "FALSE", "FALSE", "TRUE"))) |>
  select(Name, Actual, .pred_class, .pred_TRUE) |>
  rename(Predicted = .pred_class, Prob_HighValue = .pred_TRUE)

Marcus Lee (48 visits, high avg value, long tenure) should confidently predict TRUE. Gary McSnarl (1 visit, very low spend) should predict FALSE. Amara Diallo (recent but 14 visits already) is an interesting case — the model has to decide whether short tenure or high transaction count wins.

Exercise 8.2 — Threshold Analysis

Calculate sensitivity and specificity at thresholds of 0.30, 0.50, and 0.70. Present the results in a table. Which threshold would you recommend for FreshMart’s loyalty campaign, and why?

✅ Solution 
map(c(0.30, 0.50, 0.70), function(t) {
  lr_preds |>
    mutate(.pred_adj = as.factor(if_else(.pred_TRUE >= t, "TRUE", "FALSE"))) |>
    metric_set(sensitivity, specificity)(truth = IsHighValue, estimate = .pred_adj) |>
    mutate(threshold = t)
}) |>
  list_rbind() |>
  select(threshold, .metric, .estimate) |>
  pivot_wider(names_from = .metric, values_from = .estimate)

For a loyalty campaign where missing a high-value customer is costly, a threshold of 0.30 maximises sensitivity. The cost is more false positives — offers sent to regular customers — which FreshMart may consider acceptable if the campaign cost per customer is low.

Exercise 8.3 — The Naive Baseline

The simplest possible classifier always predicts FALSE (the majority class). Calculate its accuracy, sensitivity, and specificity on the test set. How does logistic regression compare?

(Hint: create a column .pred_naive = factor("FALSE", levels = c("FALSE", "TRUE")) and run metric_set(accuracy, sensitivity, specificity) on it.)

✅ Solution 
naive_preds <- lr_preds |>
  mutate(.pred_naive = factor("FALSE", levels = c("FALSE", "TRUE")))

metric_set(accuracy, sensitivity, specificity)(
  naive_preds, truth = IsHighValue, estimate = .pred_naive
)

The naive classifier has sensitivity = 0 — it catches none of the high-value customers. Despite potentially high accuracy, it is completely useless for the loyalty team’s purpose. Logistic regression dramatically improves sensitivity.
